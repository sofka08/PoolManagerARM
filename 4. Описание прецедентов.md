# Описание прецедентов #
### Прецедент №1: Аутентификация пользователя ###
| Раздел         | Описание                                                                 |
|----------------|--------------------------------------------------------------------------|
| ID             | UC-01                                                                   |
| Название       | Аутентификация пользователя в системе                                     |
| Цель           | Получить безопасный доступ к функционалу АРМ согласно роли (Администратор или Тренер). |
| Актор          | Администратор / Тренер                                                   |
| Предварительные условия | 1. Приложение запущено. <br>2. В базе данных SQLite существуют учетные записи пользователей. |
| Основной поток  | 1. Пользователь вводит Логин и Пароль. <br>2. Система проверяет данные в таблице Users. <br>3. При совпадении данных определяется роль (Admin/Coach). <br>4. Открывается главное меню с доступным набором функций. |
| Исключения     | • Неверные данные: Вывод ошибки «Неверный логин или пароль». <br>• Пустые поля: Блокировка входа до заполнения всех полей. |
| Постусловия    | Пользователь авторизован и получил доступ к интерфейсу согласно своим правам. |  

### Прецедент №2: Управление клиентской базой ###
| Раздел         | Описание                                                                 |
|----------------|--------------------------------------------------------------------------|
| ID             | UC-02                                                                   |
| Название       | Управление данными клиентов                                                 |
| Цель           | Ведение актуального реестра посетителей бассейна.                               |
| Актор          | Администратор                                                           |
| Основной поток  | 1. Переход в раздел «Клиенты». <br>2. Поиск клиента по ФИО или номеру телефона. <br>3. Выполнение CRUD-операций: добавление нового, редактирование профиля или удаление. <br>4. Просмотр истории посещений и активных абонементов клиента. |
| Исключения     | • Дубликат: Попытка регистрации клиента с уже существующим в базе номером телефона. <br>• Ошибки ввода: ввод некорректных символов в поле «Телефон». |
| Постусловия    | База данных клиентов обновлена и сохранена в SQLite.                          |

###  Прецедент №3: Управление абонементами и их продажа ###
| Раздел         | Описание                                                                 |
|----------------|--------------------------------------------------------------------------|
| ID             | UC-03                                                                   |
| Название       | Управление типами абонементов и продажи                                      |
| Цель           | Контроль номенклатуры услуг и регистрация фактов продаж.                         |
| Актор          | Администратор                                                           |
| Основной поток  | 1. Создание нового типа абонемента (цена, срок, кол-во занятий). <br>2. Привязка абонемента к выбранному клиенту. <br>3. Автоматический расчет даты окончания действия абонемента. |
| Исключения     | • Удаление: Запрет на удаление типа абонемента, если он привязан к активным клиентам. |
| Постусловия | Информация о продаже зафиксирована в финансовой отчетности и профиле клиента. |    

###  Прецедент №4: Управление тренерским составом ###
| Раздел         | Описание                                                                 |
|----------------|--------------------------------------------------------------------------|
| ID             | UC-04                                                                   |
| Название       | Управление данными о тренерах                                                 |
| Цель           | Вести учет сотрудников-тренеров и их учетных записей для доступа к АРМ.               |
| Актор          | Администратор                                                           |
| Предварительные условия | 1. Администратор авторизован в системе. <br>2. Открыт модуль "Тренеры". |
| Основной поток  | 1. Администратор видит список всех зарегистрированных тренеров. <br>2. Добавление: Нажимает "Добавить нового тренера", указывает ФИО, специализацию, телефон, стаж. <br>3. Редактирование: Выбирает тренера, изменяет его данные или учетную запись. <br>4. Удаление: Выбирает тренера, подтверждает удаление. <br>5. После любого действия, список тренеров обновляется. |
| Исключения | • Удаление занятого тренера: Если тренер имеет активные записи в расписании, система блокирует удаление с уведомлением. <br>• Попытка создать дубликат логина: Система выдает ошибку, если логин уже занят. |
| Постусловия    | Список тренеров актуализирован. Учетные записи тренеров созданы/изменены/удалены. |

### Прецедент №5: Планирование и просмотр расписания ###
| Раздел         | Описание                                                                 |
|----------------|--------------------------------------------------------------------------|
| ID             | UC-05                                                                   |
| Название       | Планирование занятий                                                       |
| Цель           | Резервирование времени и дорожек для проведения тренировок.                   |
| Актор          | Администратор / Тренер                                                   |
| Основной поток  | 1. Выбор даты на интерактивном календаре. <br>2. Выбор свободного временного слота. <br>3. Назначение тренера и выбор клиента из базы. <br>4. Проверка системы на отсутствие пересечений («двойного бронирования»). <br>5. Сохранение записи. |
| Исключения     | • Конфликт времени: Выбранный тренер или дорожка уже заняты в этот период – система блокирует запись. |
| Постусловия    | Запись отображается в общем календаре расписания.                          |

### Прецедент №6: Просмотр статистики и отчетности ###
| Раздел         | Описание                                                                 |
|----------------|--------------------------------------------------------------------------|
| ID             | UC-06                                                                   |
| Название       | Просмотр аналитических данных                                                 |
| Цель           | Получение отчетов о финансовой деятельности и загруженности объекта.               |
| Актор          | Администратор                                                           |
| Основной поток  | 1. Выбор периода (начальная и конечная даты). <br>2. Генерация отчета по выручке. <br>3. Просмотр лидеров среди тренеров по количеству занятий. |
| Исключения     | • Нет данных: Выбран период, в котором не было совершено ни одной операции. |
| Постусловия    | Информация представлена в виде таблиц или графиков на экране.                 |

###  Прецедент №7: Мониторинг текущей загрузки бассейна ###
| Раздел         | Описание                                                                 |
|----------------|--------------------------------------------------------------------------|
| ID             | UC-07                                                                   |
| Название       | Контроль текущей заполняемости                                                  |
| Цель           | Визуализация количества человек в чаши бассейна в реальном времени.               |
| Актор          | Администратор / Тренер                                                   |
| Основной поток  | 1. Открытие модуля «Загрузка». <br>2. Система считывает активные записи из расписания на текущую минуту. <br>3. Вывод количества человек по зонам (например: Дорожка 1, VIP-зона). |
| Исключения     | • Нет активных сессий: Если ни в одной зоне нет активности, система отображает "Активных сессий: 0" |
| Постусловия    | Персонал владеет информацией для соблюдения норм безопасности.                 |
