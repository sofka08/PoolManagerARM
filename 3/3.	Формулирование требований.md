# 3. Формулирование требований к системе

## 3.1. Функциональные требования

Функциональные требования описывают конкретные действия, которые пользователь может совершать в системе.   
В данной версии АРМ реализовано полное разделение прав доступа между **Администратором** и **Тренером**.

Программный продукт должен обеспечивать выполнение следующих функций:
*   **Идентификация пользователей:** Система должна запрашивать уникальный логин и пароль при запуске.
*   **Разграничение прав:** Автоматическая настройка интерфейса в зависимости от роли (Admin/Coach).
*   **Управление клиентами**
    *   **Регистрация:** Добавление новых клиентов с указанием ФИО, даты рождения, контактного телефона.
    *   **Редактирование:** Возможность обновления персональных данных при их изменении.
    *   **Поиск и фильтрация:** Мгновенный поиск по фамилии или номеру телефона.
    *   **История посещений:** Отображение всех когда-либо купленных абонементов и посещенных занятий конкретным клиентом.
    *   **Удаление:** Безвозвратное удаление профиля клиента из системы (доступно только Администратору).
*   **Абонементы и продажи**
    *   **Каталог услуг:** Создание различных типов абонементов (разовые, на 8 занятий, безлимит на месяц и т.д.).
    *   **Управление ценами:** Установка и изменение стоимости услуг.
    *   **Оформление продаж:** Привязка выбранного абонемента к профилю клиента с фиксацией даты сделки.
    *   **Контроль статуса:** Автоматическое определение состояния абонемента (Активен / Истек / Исчерпан лимит занятий).
    *   **Поиск:** Мгновенный поиск по названию абонемента или количеству занятий.
*   **Управление персоналом (Тренеры)**
    *   **Регистрация и редактирование:** Добавление новых сотрудников с указанием полной информации и возможность ее редактирования.
    *   **Поиск и фильтрация:** Мгновенный поиск по фамилии или номеру телефона.
    *   **Список тренеров:** Ведение списка сотрудников с указанием их специализации, стажа, даты рождения и прочей информации.
    *   **Связь с расписанием:** Возможность назначать конкретного тренера на определенный временной слот и отслеживать график каждого сотрудника.
*   **Интерактивное расписание**
    *   **Календарное отображение:** Просмотр сетки занятий на выбранный день/неделю.
    *   **Бронирование:** Создание записи на занятие с выбором клиента, тренера и зоны бассейна.
    *   **Контроль конфликтов:** Система должна блокировать сохранение записи, если выбранный тренер или зона бассейна уже заняты на это время.
    *   **Отмена/Перенос:** Возможность оперативно изменить время занятия или удалить запись.
*   **Аналитика и мониторинг**
    *   **Загрузка бассейна:** Визуальное отображение текущего количества человек в каждой зоне бассейна в реальном времени.
    *   **Финансовая статистика:** Генерация отчетов по выручке за выбранный период (день, месяц, год).
    *   **Рейтинг услуг:** Анализ самых популярных типов абонементов среди клиентов.
    *   **Отчет по тренерам:** Подсчет количества проведенных занятий каждым сотрудником.
   
## 3.2. Нефункциональные требования
Нефункциональные требования определяют качественные характеристики работы системы.

### Производительность
*   Время запуска приложения и первичной загрузки данных из БД SQLite не должно превышать **2 секунд**.
*   Поиск по базе данных должен осуществляться практически мгновенно (отклик до **0.5 секунд**).
*   Обновление графического интерфейса при переключении между вкладками должно происходить без видимых задержек.

### Надежность и сохранность данных
*   **Целостность данных:** Применение механизмов внешних ключей (Foreign Keys) в БД, чтобы нельзя было удалить клиента, у которого есть активные записи в расписании.
*   **Устойчивость к сбоям:** Система должна корректно завершать транзакции при внезапном закрытии приложения.
*   **Надежность:** Автоматическое сохранение данных в SQLite при закрытии любой формы.

### Удобство использования
*   **Интуитивность:** Интерфейс должен быть понятен пользователю с минимальными навыками владения ПК.
*   **Навигация:** Наличие панели быстрого доступа к основным модулям и кнопка «Выход».

### Безопасность
*   Запрет на выполнение административных функций (удаление, статистика) для пользователей с ролью «Тренер».
*   Блокировка доступа к файлу базы данных через интерфейс приложения для неавторизованных лиц.
