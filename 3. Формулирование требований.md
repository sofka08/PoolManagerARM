# 3. Формулирование требований к системе
## 3.1. Функциональные требования

Функциональные требования описывают конкретные действия, которые пользователь может совершать в системе.   
В данной версии АРМ реализовано полное разделение прав доступа между **Администратором** и **Тренером**.

**1. Авторизация и разграничение доступа**
*   Вход в систему по идентификационным данным (логин/пароль).
*   Предоставление прав доступа согласно роли (Администратор/Тренер).

**2. Ведение базы клиентов**
*   Регистрация новых клиентов и редактирование профилей.
*   Поиск и фильтрация клиентов в реестре.
*   Просмотр персональной истории посещений и покупок.
*   Удаление учетных записей (только для роли Администратор).
**3. Учет абонементов и регистрация продаж**
*   Формирование каталога услуг и типов абонементов.
*   Установление стоимости и параметров услуг.
*   Оформление сделок (продажа абонемента клиенту).
*   Мониторинг статуса и лимитов активных абонементов.

**4. Сопровождение тренерского состава**
*   Ведение реестра тренеров и их специализаций.
*   Назначение ответственных сотрудников на занятия.
*   Контроль персонального рабочего графика тренеров.

**5. Организация и контроль расписания**
*   Формирование сетки занятий и бронирование дорожек.
*   Автоматическая проверка конфликтов (пересечение времени/ресурсов).
*   Оперативная корректировка (перенос/отмена) записей.

**6. Мониторинг и аналитика**
*   Контроль текущей загрузки зон бассейна в реальном времени.  
*   Генерация финансовой отчетности за выбранные периоды.
*   Формирование статистики популярности услуг и продуктивности персонала.  

## 3.2. Нефункциональные требования
Нефункциональные требования определяют качественные характеристики работы системы.

### Производительность
*   Время запуска приложения и первичной загрузки данных из БД SQLite не должно превышать **2 секунд**.
*   Поиск по базе данных должен осуществляться практически мгновенно (отклик до **0.5 секунд**).
*   Обновление графического интерфейса при переключении между вкладками должно происходить без видимых задержек.

### Надежность и сохранность данных
*   **Целостность данных:** Применение механизмов внешних ключей (Foreign Keys) в БД, чтобы нельзя было удалить клиента, у которого есть активные записи в расписании.
*   **Устойчивость к сбоям:** Система должна корректно завершать транзакции при внезапном закрытии приложения.
*   **Надежность:** Автоматическое сохранение данных в SQLite при закрытии любой формы.

### Удобство использования
*   **Интуитивность:** Интерфейс должен быть понятен пользователю с минимальными навыками владения ПК.
*   **Навигация:** Наличие панели быстрого доступа к основным модулям и кнопка «Выход».

### Безопасность
*   Запрет на выполнение административных функций (удаление, статистика) для пользователей с ролью «Тренер».
*   Блокировка доступа к файлу базы данных через интерфейс приложения для неавторизованных лиц.
